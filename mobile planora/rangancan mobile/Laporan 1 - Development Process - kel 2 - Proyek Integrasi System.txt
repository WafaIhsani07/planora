Bagian ini menguraikan analisis kebutuhan dan alur pengalaman pengguna (UX) yang dikhususkan untuk Web Dashboard Planora. Sesuai dengan batasan masalah sistem, web dashboard ini dirancang secara spesifik untuk memfasilitasi dua peran utama, yaitu Vendor (penyedia jasa event) dan Admin Sistem (pengelola platform). Berbeda dengan aplikasi mobile yang mengutamakan kemudahan transaksi pelanggan, web dashboard mengutamakan efisiensi operasional dan kemudahan pengelolaan data.
UX Requirements Web Dashboard
Berdasarkan analisis kebutuhan non-fungsional sistem yang menuntut aplikasi agar user-friendly dan memiliki antarmuka yang jelas, perancangan web dashboard Planora untuk Vendor dan Admin harus memenuhi beberapa kriteria kebutuhan UX berikut:
Kebutuhan Efisiensi Manajemen (Operational Efficiency)
Vendor dan Admin memiliki tugas operasional yang padat setiap harinya. Dashboard harus mampu menyajikan informasi terpenting secara langsung tanpa pengguna perlu menelusuri banyak halaman terlebih dahulu.
Penerapan
Halaman utama dashboard wajib menampilkan ringkasan operasional harian dalam bentuk Summary Cards yang mudah dibaca secara sekilas (at-a-glance). Bagi Vendor, kartu ini menampilkan jumlah pesanan masuk hari ini, jadwal layanan yang akan datang, dan pembayaran yang masih menunggu verifikasi. Bagi Admin, kartu ini menampilkan jumlah vendor baru yang menunggu verifikasi, total transaksi aktif, dan laporan atau keluhan pengguna yang belum ditangani.
Kebutuhan Pengelolaan Konten Layanan (Content Management)
Vendor harus mampu mengelola seluruh informasi layanan mereka secara mandiri melalui dashboard, tanpa memerlukan bantuan teknis dari tim pengembang.
Penerapan
Halaman pengelolaan layanan dirancang dengan form input yang intuitif untuk menambah, mengubah, atau menonaktifkan paket layanan. Setiap field input dilengkapi validasi real-time yang langsung memberikan feedback apabila data yang 
dimasukkan tidak valid (misalnya: harga harus berupa angka, nama paket tidak boleh kosong). Sebelum penghapusan data layanan, sistem menampilkan dialog konfirmasi agar vendor tidak secara tidak sengaja menghapus data penting yang masih aktif digunakan.
Kebutuhan Manajemen Jadwal & Konfirmasi Pesanan (Schedule & Order Management)
Vendor perlu memantau seluruh pesanan yang masuk dan mengelola ketersediaan jadwal mereka secara efisien agar tidak terjadi tumpang tindih layanan.
Penerapan
Halaman manajemen pesanan menampilkan daftar pesanan masuk secara real-time disertai detail pelanggan, layanan yang dipesan, tanggal acara, dan status pembayaran. Vendor dapat mengonfirmasi atau menolak pesanan langsung dari halaman ini dengan satu klik, disertai isian catatan alasan penolakan apabila pesanan ditolak. Sistem secara otomatis memperbarui ketersediaan jadwal pada kalender vendor setiap kali pesanan dikonfirmasi, sehingga slot tersebut tidak bisa dipesan lagi oleh pelanggan lain melalui aplikasi mobile.
Kebutuhan Rekap Transaksi & Laporan Keuangan (Financial Visibility)
Vendor membutuhkan visibilitas yang jelas terhadap arus keuangan layanan mereka, sementara Admin membutuhkan akses menyeluruh terhadap seluruh transaksi yang terjadi di platform.
Penerapan
Halaman laporan transaksi menampilkan riwayat pesanan yang dapat difilter berdasarkan status pembayaran (Belum Bayar, DP, Lunas) maupun rentang tanggal tertentu. Vendor dapat melihat rekap total pendapatan per periode secara langsung. Halaman ini juga menampilkan preview thumbnail bukti transfer yang diunggah pelanggan, sehingga vendor dapat memverifikasi pembayaran tanpa harus mengunduh file terlebih dahulu. Bagi Admin, tersedia halaman monitoring seluruh transaksi lintas vendor dengan kemampuan filter dan pencarian data yang komprehensif.
Kebutuhan Alur Verifikasi Vendor yang Efisien (Vendor Verification Workflow)
Admin bertanggung jawab memastikan setiap vendor yang beroperasi di platform Planora telah terverifikasi kredibilitasnya agar pelanggan terhindar dari penipuan.
Penerapan
Halaman verifikasi vendor menampilkan daftar pendaftaran vendor baru yang menunggu persetujuan. Setiap entri menampilkan detail profil usaha, dokumen pendukung, dan galeri portofolio yang dapat diperiksa langsung dari dashboard. Admin dapat menyetujui atau menolak pendaftaran dengan satu klik. Apabila ditolak, Admin diwajibkan mengisi catatan alasan penolakan yang akan dikirimkan sebagai notifikasi kepada vendor terkait, sehingga vendor dapat melakukan perbaikan dan mendaftar ulang.
Kebutuhan
 Keamanan Akses 
Berbasis
 Peran (Role-Based Secure Access)
Web dashboard digunakan oleh dua peran dengan kewenangan yang sangat berbeda. Sistem harus memastikan setiap pengguna hanya dapat mengakses data dan fitur sesuai peran mereka.
Penerapan
Login web dashboard menggunakan kredensial aman dengan mekanisme JWT Token yang memiliki masa berlaku terbatas. Setiap sesi login secara otomatis terputus setelah periode tidak aktif tertentu. Data yang ditampilkan pada dashboard vendor dibatasi secara ketat hanya pada layanan, jadwal, pesanan, dan transaksi milik vendor yang bersangkutan (data isolation per vendor). Admin memiliki akses penuh ke seluruh data platform namun tidak dapat mengubah data layanan atau transaksi milik vendor secara langsung, melainkan hanya dapat mengelola status verifikasi dan akun pengguna.
UX Flow Web Dashboard
Profil Pengguna (User Persona) & Pendekatan Desain
Sebelum merancang alur UX, perlu dipahami karakteristik dua target pengguna utama dari web dashboard ini:
Vendor (
Penyedia
 Jasa Event)
Vendor merupakan pemilik atau pengelola usaha jasa event skala kecil hingga menengah seperti fotografer, katering, penyedia venue, atau event organizer. Karakteristik pengguna ini umumnya:
Tidak selalu memiliki latar belakang teknis, sehingga membutuhkan antarmuka yang sederhana dan mudah dipahami tanpa pelatihan khusus.
Mengutamakan kecepatan dalam mengonfirmasi pesanan dan memverifikasi pembayaran agar operasional harian tidak terganggu.
Membutuhkan visibilitas jadwal yang jelas agar tidak terjadi tumpang tindih layanan yang dapat merugikan reputasi usaha mereka.
Admin Sistem
Admin merupakan pengelola platform Planora yang bertanggung jawab atas keberlangsungan dan kualitas ekosistem marketplace. Karakteristik pengguna ini umumnya:
Membutuhkan akses cepat dan terpusat terhadap seluruh aktivitas platform dalam satu layar monitoring.
Memerlukan alur kerja yang efisien untuk menangani verifikasi vendor dan keluhan pengguna dalam jumlah banyak.
Mengutamakan kemampuan pencarian dan filter data yang kuat untuk menemukan informasi spesifik dengan cepat.
Oleh karena itu, prinsip utama UX web dashboard Planora adalah Efisiensi Operasional (Efficiency) dan Kontrol Data yang Jelas (Clarity of Control) agar memenuhi standar kebutuhan non-fungsional sistem yang user-friendly dan memiliki performa stabil.

Penanganan Kendala (Error Handling) pada UX
Untuk mencegah terjadinya kesalahan dalam proses verifikasi vendor, sistem merespons interaksi Admin sebagai berikut:
Skenario Normal: Apabila data vendor dinilai lengkap dan valid, Admin menekan tombol ‘Setujui’ dan sistem langsung mengaktifkan akun vendor serta menampilkan layanan mereka di platform.
Skenario Penolakan: Apabila data tidak lengkap atau mencurigakan, tombol ‘Kirim Penolakan’ hanya dapat ditekan setelah field catatan alasan diisi. Hal ini memastikan vendor selalu mendapatkan informasi yang jelas mengenai alasan penolakan sehingga dapat melakukan perbaikan.
UX Interaction Flow: Proses Konfirmasi & Verifikasi Pembayaran (Vendor)
Dikarenakan sistem Planora menggunakan transfer manual, alur verifikasi pembayaran oleh Vendor di web dashboard harus dirancang se-efisien mungkin agar proses konfirmasi tidak memakan waktu lama dan pesanan pelanggan dapat segera diproses.
Analisis Interaksi Verifikasi Pembayaran
Pada flow ini, fitur preview thumbnail bukti transfer langsung di halaman verifikasi menjadi elemen UX yang sangat krusial. Alih-alih memaksa vendor mengunduh file bukti terlebih dahulu, tampilan preview langsung mempersingkat waktu verifikasi secara signifikan. Pemisahan yang jelas antara tombol ‘Verifikasi Lunas’ dan ‘Tolak Pembayaran’ dengan warna berbeda (hijau dan merah) juga mengurangi risiko vendor menekan tombol yang salah saat memproses banyak transaksi secara bersamaan.

UX Navigation Structure (Sitemap Web Dashboard)
Diagram berikut menunjukkan struktur peta navigasi antarmuka (UX Navigation Map) pada web dashboard Planora untuk kedua peran pengguna: Vendor dan Admin.
Analisis Navigasi
Struktur navigasi web dashboard Planora dirancang berdasarkan prinsip pemisahan peran yang tegas (role separation). Dashboard Vendor dan Dashboard Admin memiliki hierarki menu yang sepenuhnya terpisah meskipun berada dalam satu platform web yang sama. Pemisahan ini penting karena kedua peran memiliki tanggung jawab dan kebutuhan informasi yang sama sekali berbeda, sehingga mencampurkan keduanya dalam satu tampilan akan menyebabkan cognitive overload dan potensi kesalahan operasional. Vendor hanya melihat data dan menu yang relevan dengan layanan mereka sendiri, sementara Admin memiliki akses ke seluruh data platform namun dalam konteks monitoring dan manajemen, bukan transaksi langsung.
